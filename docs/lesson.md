## 0. ปรัชญาเริ่มต้น

โปรแกรมคำนวณวิศวกรรม (สำรวจ/ถนน/สะพาน) ต่างจากโปรแกรมทั่วไปตรงที่ **"ผิดเป็นเมตรไม่ได้"**
หนึ่งบรรทัดที่คำนวณพลาดอาจกลายเป็นเสาเข็มผิดตำแหน่ง ดังนั้นหลักคิดทั้งหมดในเอกสารนี้
มุ่งไปที่สามคำ: **ถูกต้อง (correct) → ตรวจสอบได้ (verifiable) → แก้ไขต่อยอดได้ (maintainable)**

กฎเหล็กที่เราใช้มาตลอดโปรเจกต์:
- **คำนวณด้วยความละเอียดเต็มที่ภายใน ปัดเศษเฉพาะตอนแสดงผล**
- **เขียนเป็นฟังก์ชันบริสุทธิ์ (pure function) แยก "การคำนวณ" ออกจาก "การรับ/แสดงข้อมูล"**
- **ทุกครั้งที่เพิ่ม/แก้ ต้องมีชุดทดสอบยืนยันว่าของเดิมไม่พัง**

## 1. Software Architecture — สถาปัตยกรรมซอฟต์แวร์

### (ก) แนวคิด: แบ่งเป็นชั้น (Layered) และให้ "แกนคำนวณ" บริสุทธิ์

สถาปัตยกรรมที่ดีคือการตอบคำถามว่า *"อะไรพึ่งอะไร และใครห้ามรู้จักใคร"*
เราใช้รูปแบบ **Layered Architecture** + **Functional Core / Imperative Shell**
(แกนกลางเป็นคณิตศาสตร์ล้วน ขอบนอกเป็นการต่อกับโลกภายนอก)

```
            ┌─────────────────────────────────────────────┐
   ชั้นนอก   │  INTERFACE / ADAPTER (ขอบนอก, "เปลือก")        │
  (I/O)     │  - UDF ใน Google Sheets / Excel VBA           │
            │  - ปุ่ม Build / Check                          │
            │  - อ่านช่วงเซลล์ → ป้อนแกน → จัดรูปผลลัพธ์         │
            └───────────────▲─────────────────────────────┘
                            │ (เรียกลงล่างเท่านั้น)
            ┌───────────────┴─────────────────────────────┐
   ชั้นกลาง  │  BUILDERS (ประกอบงานเป็นชุด)                    │
            │  - AlignmentBuilder: PI → ตาราง element        │
            │  - VerticalBuilder : VPI → ตารางโค้งดิ่ง         │
            └───────────────▲─────────────────────────────┘
                            │
            ┌───────────────┴─────────────────────────────┐
   ชั้นโดเมน │  DOMAIN ENGINE (ตรรกะงานสำรวจ, บริสุทธิ์)        │
            │  - Alignment (element, stationToCoord ...)    │
            │  - Vertical / CrossFall / Surface3D           │
            └───────────────▲─────────────────────────────┘
                            │
            ┌───────────────┴─────────────────────────────┐
   ชั้นฐาน   │  FOUNDATION (คณิตศาสตร์ทั่วไป, บริสุทธิ์ที่สุด)   │
            │  - FPMath (มุม/ปัดเศษ), WCB (polar↔rectangular)│
            └─────────────────────────────────────────────┘
```

### (ข) เห็นในโปรเจกต์เราตรงไหน
- `FWD_N` (UDF) เป็นแค่ **adapter บางๆ**: อ่านช่วงเซลล์ → `parseTable_` → เรียก
  `Alignment.stationToCoord` → คืนตัวเลข มันไม่รู้จักวิธีคำนวณโค้งเลย
- `Alignment.stationToCoord` (โดเมน) ไม่รู้จัก "เซลล์" หรือ "Excel" — รับ array of element + ตัวเลข คืนตัวเลข
- เพราะแกนไม่รู้จัก UI เราจึงเอา engine ตัวเดียวไปวางได้ทั้ง **Apps Script UDF, VBA UDF, และปุ่ม** = *Single Source of Truth*

### (ค) หลักการกลาง (ทุกภาษา)
1. **กฎทิศทางพึ่งพา (Dependency Rule):** ชั้นในห้ามรู้จักชั้นนอก ลูกศรพึ่งพาชี้เข้าด้านในเสมอ
   แกนคำนวณต้องคอมไพล์/รันได้โดยไม่ต้องมี UI
2. **Functional Core, Imperative Shell:** ตรรกะคณิตศาสตร์ = ฟังก์ชันบริสุทธิ์ (ทดสอบง่าย),
   ส่วนที่ "สกปรก" (อ่านไฟล์/เซลล์/เครือข่าย/แสดงผล) ดันออกไปไว้ขอบนอก
3. **Single Source of Truth:** ตรรกะหนึ่งเรื่องเขียนที่เดียว แล้วให้ทุกหน้าจอ/แพลตฟอร์มเรียกใช้
   ถ้าพบว่าต้องแก้สูตรเดียวกันสองที่ — แสดงว่าสถาปัตยกรรมผิด
4. **Stable boundary, flexible inside:** *หน้าตา* ของฟังก์ชันสาธารณะ (ชื่อ/พารามิเตอร์) ควรนิ่ง
   ส่วน *ภายใน* ปรับปรุงได้อิสระ

## 2. Logic — ตรรกะการเขียนโปรแกรม

### 2.1 เลือก "ตัวแทนข้อมูล" ให้ดีก่อน แล้วโค้ดจะง่ายเอง

**(ก) แนวคิด:** ปัญหาที่ดูซับซ้อนมักยุบรวมได้ ถ้าเลือก *data model* ถูก

**(ข) ในโปรเจกต์:** แทนที่จะแยกเคส "ทางตรง / โค้งวงกลม / โค้งคลายตัว" เป็นโค้ดสามชุด
เราใช้ค่าเดียวคือ **ความโค้ง k = 1/R** (มีเครื่องหมาย: + เลี้ยวขวา, − เลี้ยวซ้าย, ทางตรง k = 0)
แล้วทุกชิ้นกลายเป็น element เดียวกันที่มี `kIn, kOut`:

| ชนิด | kIn | kOut |
|---|---|---|
| ทางตรง (tangent) | 0 | 0 |
| โค้งวงกลม (circular) | k | k |
| โค้งคลายตัว (spiral) | k₁ | k₂ |

> *"ทำให้สถานะที่ผิดกฎเป็นไปไม่ได้ (make illegal states unrepresentable)"* — เมื่อเลือกตัวแทนดี
> เคสพิเศษจะหายไปกลายเป็นค่าขอบ (k=0 คือทางตรง) ไม่ต้องเขียน if แยกเต็มไปหมด

**(ค) หลักการกลาง:** ออกแบบ struct/record/object ที่จับ "แก่น" ของปัญหา ก่อนเขียนตรรกะ
ตัวแทนที่ดี = โค้ดสั้นลง บั๊กน้อยลง

### 2.2 เครื่องสถานะและการส่งต่อสถานะ (State forwarding)

**(ข) ในโปรเจกต์:** ปลายของ element ก่อนหน้า = ต้นของ element ถัดไป
(`Exit State(n) = Entry State(n+1)`) แนวทั้งเส้นจึงต่อเนื่องโดยไม่มีรอยต่อ และตรวจสอบได้ทีละช่วง

**(ค) หลักการกลาง:** กำหนด **invariant** (สิ่งที่ต้องจริงเสมอ) เช่น "ทิศปลายช่วงนี้ = ทิศต้นช่วงหน้า"
แล้วเขียนโค้ดให้รักษา invariant นั้น การมี invariant ชัดเจนทำให้ดีบักง่าย

### 2.3 แยก "คำนวณ" ออกจาก "จัดรูปแบบ"

**(ข) ในโปรเจกต์:** engine คืนเรเดียน/ตัวเลขเต็มความละเอียด ส่วนการปัดเป็น 4 ตำแหน่ง
หรือแปลงเป็น DMS ทำเฉพาะตอน "แสดง" เท่านั้น

**(ค) หลักการกลาง:** *compute ≠ format* อย่าปัดเศษกลางทาง เพราะค่าที่ปัดแล้วเอาไปคำนวณต่อจะสะสมความคลาด

### 2.4 ป้องกันที่ขอบ ล้มให้ดังและชัด (Guard clause / Fail fast)

**(ข) ในโปรเจกต์:** ก่อนสร้างแนว เราตรวจว่าจุดควบคุมไม่มีรัศมี (ถูกวางเป็น PI ผิด) แล้วหยุดพร้อม
ข้อความไทยที่อ่านรู้เรื่อง แทนที่จะปล่อยให้ไหลไปเป็น `NaN` ลึกๆ จนหาไม่เจอ

```
function buildFromPI(pis):
    if some PI has no radius where required:
        throw Error("จุด ... ไม่มีรัศมี — ตรวจคอลัมน์ Radius")   # ล้มที่ขอบ ชัดเจน
    ... (ภายในนี้สมมุติว่าข้อมูลถูกต้องแล้ว) ...
```

**(ค) หลักการกลาง:**
- ตรวจ input ที่ "ขอบนอก" (boundary) ให้ครบ แล้วภายในแกนถือว่าข้อมูลสะอาด → โค้ดในแกนสั้นและมั่นใจได้
- ข้อความ error ต้องบอก *อะไรผิด + ตรงไหน + แก้ยังไง*
- คืน "ค่าบอกข้อผิดพลาด" ที่ชัด (`#STA_OUT`, `#NO_ALIGN`) ดีกว่าโยน 0 หรือค่าเดาๆ ออกไปเงียบๆ

### 2.5 ฟังก์ชันบริสุทธิ์ + ผลลัพธ์ทำซ้ำได้

**(ค) หลักการกลาง:** ฟังก์ชันคำนวณควร
*(1)* input เดิม → output เดิมเสมอ (deterministic)
*(2)* ไม่แอบแก้ตัวแปรนอกตัว (no side effect)
*(3)* เรียกซ้ำได้ผลเท่าเดิม (idempotent)
สามข้อนี้ทำให้ "เขียนเทสต์ได้" และ "ย้ายภาษาได้" ซึ่งคือเหตุผลที่เราพอร์ต JS → Python (มิเรอร์) → VBA ได้

### 2.6 สัญญาเรื่องเครื่องหมาย/หน่วย ต้องเขียนให้ชัด

**(ข) ในโปรเจกต์:** `offset` + = ขวาของทิศเดินทาง, − = ซ้าย; `Radius` + = เลี้ยวขวา, − = เลี้ยวซ้าย;
มุมภายใน = เรเดียน, ที่ขอบ = องศา/DMS

**(ค) หลักการกลาง:** ความกำกวมเรื่องหน่วยและเครื่องหมายคือต้นเหตุบั๊กคลาสสิก
เขียน "สัญญา (contract)" ไว้ในคอมเมนต์/เอกสาร และ *รักษาให้สม่ำเสมอทั้งระบบ*

## 3. Concept — SAFE + SMALL + STABLE + MODULAR

หลักสี่คำนี้คือ "เข็มทิศ" ของโปรเจกต์ ทุกครั้งที่ตัดสินใจออกแบบ ให้ถามว่าเลือกทางไหนแล้ว
**ปลอดภัยขึ้น เล็กลง นิ่งขึ้น และเป็นโมดูลมากขึ้น** หรือไม่

### 3.1 SAFE — ปลอดภัย
- ไม่เชื่อ input: ตรวจที่ขอบ, ปัดเคสว่าง/ผิดชนิดทิ้ง (เช่นข้ามแถวหัวตารางด้วย `isNumeric`)
- ไม่ล้มเงียบ: ผิดแล้วคืนรหัสที่ชัด ไม่คืนค่าเดา
- ป้องกันหารด้วยศูนย์/ค่าใกล้ศูนย์ (เช็ก `k = 0` ก่อนคิด `1/k`)
- เทียบทศนิยมด้วย *ค่าเผื่อ (tolerance)* ไม่ใช้ `==` ตรงๆ
- **ทดสอบทางคณิตศาสตร์:** เปรียบเทียบ forward→inverse ต้องคืนค่าเดิม (roundtrip = 0)

### 3.2 SMALL — เล็ก
- ฟังก์ชันหนึ่งทำงานเดียว สั้นพอที่อ่านจบใน "หนึ่งหน้าจอ"
- ความซับซ้อนต่ำ (cyclomatic complexity น้อย → if ซ้อนน้อย) ทดสอบง่าย
- ประกอบจากชิ้นเล็ก: `pointOnElement` เล็ก → `stationToCoord` ใช้มันต่อ → `FWD_N` ใช้ต่ออีกที
- ของเล็กแก้ง่าย ย้ายภาษาง่าย และพิสูจน์ความถูกต้องได้ทีละชิ้น

### 3.3 STABLE — นิ่ง/เสถียร
- **เสถียรเชิงผลลัพธ์:** input เดิม → output เดิม (มี regression test เช่น AllTests 45/45 คอยเฝ้า)
- **เสถียรเชิงตัวเลข (numerical stability):** เลือกสูตรที่ไม่ขยายความคลาด (ดูหัวข้อ 4)
- **เสถียรเชิง API:** ชื่อ/พารามิเตอร์ของฟังก์ชันสาธารณะไม่เปลี่ยนพร่ำเพรื่อ
  (เพิ่มของใหม่แบบ *เติม* ไม่ใช่ *รื้อ* — เช่นเราเพิ่ม `azimuthAtStation` โดยไม่แตะ `stationToCoord`)
- ของเดิมที่เคยถูก ต้องถูกตลอด การเพิ่มฟีเจอร์ห้ามทำให้ของเก่าพัง

### 3.4 MODULAR — เป็นโมดูล
- ขอบเขตชัด: แต่ละโมดูลมีหน้าที่เดียว (FPMath=คณิตทั่วไป, WCB=พิกัด↔ทิศทาง, Alignment=แนวราบ)
- **เกาะกันแน่นภายใน (high cohesion), เกี่ยวพันกันน้อยภายนอก (low coupling)**
- พึ่งพากันอย่างเปิดเผย (ผ่านพารามิเตอร์/ผลลัพธ์) ไม่ใช่ผ่านตัวแปร global ลับๆ
- สลับเปลี่ยนชิ้นได้: เปลี่ยน frontend จาก Sheets เป็น Excel โดยไม่แตะแกน

> **มุมมองรวม:** SAFE คุมความถูกต้องวันนี้ · STABLE คุมความถูกต้องพรุ่งนี้ ·
> SMALL ทำให้ทั้งสองเป็นไปได้ · MODULAR ทำให้ทั้งระบบโตได้โดยไม่พันกันยุ่ง

## 4. การระวัง IEEE 754 และ Error Propagation

### 4.1 IEEE 754 double คืออะไร (ต้องเข้าใจก่อน)
ตัวเลขทศนิยมในคอมพิวเตอร์ส่วนใหญ่เป็น **binary64 (double)**: 64 บิต =
1 บิตเครื่องหมาย + 11 บิตเลขชี้กำลัง + 52 บิตเศษส่วน (mantissa)
ผลคือเก็บเลขนัยสำคัญได้ประมาณ **15–16 หลักทศนิยม** และ
**machine epsilon ≈ 2.22 × 10⁻¹⁶** (ช่องว่างเล็กสุดใกล้เลข 1)

**ข้อเท็จจริงที่ต้องจำ:** เลขฐานสิบบางตัวเก็บไม่ได้พอดีในฐานสอง เช่น `0.1`
ทำให้ `0.1 + 0.2 ≠ 0.3` (ได้ 0.30000000000000004) — *นี่ไม่ใช่บั๊กของภาษา แต่เป็นธรรมชาติของ double*

### 4.2 ผลที่ตามมา + วิธีรับมือ (เห็นในโปรเจกต์)
1. **ห้ามเทียบ float ด้วย `==`** → ใช้ค่าเผื่อ
   ในโปรเจกต์: `inRange(v, a, b, eps=1e-4)` คือเช็กว่า v อยู่ในช่วงโดยเผื่อ 0.1 มม.
   *(จำได้ไหม ตอนทดสอบ EP สถานีปัด 3 ตำแหน่งล้ำขอบ 0.2 มม. แล้ว "หาช่วงไม่เจอ" — นั่นคือผลของเรื่องนี้ตรงๆ)*
2. **การลบเลขที่ใกล้กันมาก = หายนะของความแม่น (catastrophic cancellation)**
   พิกัดสำรวจมีขนาด ~20000 ลบกันเหลือไม่กี่เมตร → นัยสำคัญหายไปหลายหลัก
   *วิธีเลี่ยง:* เก็บค่าเต็มความละเอียดภายใน, ปัดทีหลัง, และจัดลำดับการคำนวณให้ลบกันท้ายสุด
3. **ทำงานด้วยเรเดียนภายใน** แปลงเป็นองศา/DMS เฉพาะตอนแสดง — ลดการแปลงไป-กลับที่สะสมความคลาด
4. **normalize มุมด้วย floor-mod ไม่ใช่ลบทีละ 2π** (ดูหัวข้อ 5.6) เพราะการลบซ้ำๆ สะสมความคลาด

### 4.3 Error Propagation — ความคลาดเคลื่อนไหลผ่านการคำนวณอย่างไร
ทุกตัวเลขมี "ความไม่แน่นอน" ติดตัว (จากการวัด หรือจากการปัดของ double) เมื่อคำนวณต่อ
ความคลาดจะ *ไหลและขยาย* ตามชนิดการดำเนินการ:

| การดำเนินการ | ความคลาดทำตัวอย่างไร |
|---|---|
| บวก/ลบ | ความคลาด *สัมบูรณ์* บวกกัน (ลบเลขใกล้กัน → ความคลาด *สัมพัทธ์* พุ่ง) |
| คูณ/หาร | ความคลาด *สัมพัทธ์* (เปอร์เซ็นต์) บวกกัน |
| หารด้วยค่าใกล้ศูนย์ | ความคลาดพุ่งมหาศาล (ill-conditioned) |
| ฟังก์ชัน f(x) | ความคลาด ≈ |f′(x)| × (ความคลาดของ x) |

**แนวคิด condition number:** ปัญหาบางอย่าง "ไวต่อ input" มาก (ขยายความคลาด) บางอย่าง "ทน"
หน้าที่เราคือเลือก *อัลกอริทึมและลำดับการคำนวณ* ที่ทนที่สุดเท่าที่ทำได้

### 4.4 ความคลาดจากวิธีเชิงตัวเลขเอง (truncation error)
นอกจากความคลาดของ double ยังมีความคลาดจาก "วิธีประมาณ" เช่นการอินทิเกรตแบบ Simpson
มีความคลาดอันดับ **O(h⁴)** (h = ความกว้างช่วงย่อย) เราเลือก **48 ช่วง** เป็นจุดสมดุล
ระหว่างความแม่นกับความเร็ว — มากไปก็ช้าและไม่ช่วยให้แม่นเกินความละเอียดของ double อยู่ดี

### 4.5 กฎปฏิบัติ (นำไปใช้ทุกภาษา)
- ประกาศค่าเผื่อเป็น "ค่าคงที่มีชื่อ" (เช่น `TOL_STA = 1e-4`) ไม่หว่านเลขมายากลในโค้ด
- เทียบด้วย `abs(a - b) <= tol` เสมอ
- ระวังจุดที่อาจหารด้วยศูนย์ → ใส่ guard
- รู้ขนาด (scale) ของข้อมูลตัวเอง แล้วเลือกใช้ tolerance แบบสัมบูรณ์/สัมพัทธ์ให้เหมาะ
- มี "ชุดทดสอบทอง (golden tests)" ที่ค่าถูกต้องรู้แน่ๆ ไว้กันความคลาดบานปลายเงียบๆ

## 5. คณิตศาสตร์ที่นำมาใช้

### 5.1 ตรีโกณ + เรขาคณิตพิกัด (Trigonometry & Coordinate Geometry)
**Forward (polar → rectangular):** จากจุดเริ่ม (N₀,E₀) ทิศ az ระยะ d
```
N = N0 + d * cos(az)
E = E0 + d * sin(az)        # ระบบสำรวจ: az วัดจากทิศเหนือ, N=cos, E=sin
```
**Inverse (rectangular → polar):**
```
az = atan2(E2 - E1, N2 - N1)      # ใช้ atan2 ไม่ใช่ atan!
d  = sqrt((N2-N1)^2 + (E2-E1)^2)
```
> **ทำไม `atan2(y, x)` ไม่ใช่ `atan(y/x)`?** เพราะ `atan` คืนค่าได้แค่ครึ่งวง (−90°..90°)
> และพังเมื่อ x=0 ส่วน `atan2` รู้จักทั้ง 4 ควอดแรนต์และจัดการ x=0 ได้ → ได้ทิศที่ถูกเสมอ
> ในโปรเจกต์เราต้องเขียน `Atan2_` เองใน VBA เพราะ VBA ไม่มีให้ — นี่คือบทเรียนว่า
> *"ฟังก์ชันพื้นฐานบางตัวต้องสร้างเองเมื่อย้ายภาษา"*

### 5.2 เรขาคณิตเชิงวิเคราะห์ของวงกลม (Analytic Geometry)
- **คอร์ดจากส่วนโค้ง:** มุมเลี้ยว θ = k·d, ความยาวคอร์ด = `2/|k| * |sin(θ/2)|`, ทิศคอร์ด = az + θ/2
- **จุดศูนย์กลางวงกลม:** อยู่ตั้งฉากกับทิศ ระยะ R: `center = start + R*(−sin az, cos az)` ฯลฯ
- **โปรเจกต์จุดลงเส้น (perpendicular foot):** ใช้ผลคูณจุด (ดู 5.3)

### 5.3 พีชคณิตเชิงเส้น (Linear Algebra) — เวกเตอร์ + เมทริกซ์หมุน
**การหมุนพิกัด 2 มิติ (2D rotation):** สำคัญมากในการคำนวณ spiral
เราคำนวณรูปร่าง spiral ในระบบพิกัดท้องถิ่น (x ตามทิศเริ่ม, y ตั้งฉาก) แล้ว *หมุน* ด้วยมุม az ไปสู่ระบบจริง:
```
N = N0 + x*cos(az) − y*sin(az)
E = E0 + x*sin(az) + y*cos(az)
```
นี่คือเมทริกซ์หมุน `[[cos −sin],[sin cos]]` คูณเวกเตอร์ `(x, y)` นั่นเอง

**ผลคูณจุด (dot product) = การแยกองค์ประกอบตาม/ตั้งฉาก:** ตอน inverse บนทางตรง
```
ระยะตามแนว   d      = (P−Q)·t̂   = dN*cos(az) + dE*sin(az)
ระยะเยื้อง   offset = (P−Q)·n̂   = −dN*sin(az) + dE*cos(az)
```
โดย t̂ = ทิศหน่วยตามแนว, n̂ = ทิศหน่วยตั้งฉาก → พีชคณิตเชิงเส้นเปลี่ยน "เรขาคณิต" เป็น "เลขคณิต" ที่คอมพิวเตอร์ทำได้

### 5.4 แคลคูลัส + ความโค้ง (Calculus & Curvature) — หัวใจของโค้งคลายตัว
- **ความโค้ง** κ = 1/R (รัศมีน้อย = โค้งมาก)
- **ทิศทางสะสมจากความโค้ง:** θ(s) = θ₀ + ∫₀ˢ κ(u) du  (ทิศ = อินทิกรัลของความโค้งตามระยะ)
- ตำแหน่งได้จากอินทิเกรตทิศอีกชั้น: `x(s)=∫cos θ ds`, `y(s)=∫sin θ ds`
- **ชนิดโค้งคลายตัว** ต่างกันที่ "รูปร่างของความโค้ง" f(τ) เมื่อ τ=s/L:
  - Clothoid: ความโค้งเป็นเส้นตรง (∫ = τ²/2)
  - Bloss / Cosine / Sine: เส้นโค้งนุ่มกว่า (สูตร ∫ ต่างกัน)
  - **เงื่อนไขร่วม:** ทุกชนิด ∫₀¹ f = 1/2 → ปลายโค้งหมุนได้มุมเท่ากัน (ทำให้สลับชนิดได้โดยปลายตรงกัน)

### 5.5 วิธีเชิงตัวเลข (Numerical Methods)
**(1) การอินทิเกรตเชิงตัวเลข — กฎซิมป์สัน (Simpson's Rule):**
ใช้หาตำแหน่งปลาย spiral เพราะอินทิกรัลไม่มีสูตรปิด (closed form)
```
∫ ≈ (h/3) * [ f0 + 4f1 + 2f2 + 4f3 + ... + 4f(n-1) + fn ]   # n ต้องเป็นเลขคู่
```
น้ำหนัก 1-4-2-4-...-4-1 ความคลาด O(h⁴) เราใช้ n = 48

**(2) การหาราก — วิธีแบ่งครึ่ง (Bisection):**
ใช้หา "จุดเท้าตั้งฉาก" บน spiral (inverse) เพราะแก้สมการตรงๆ ไม่ได้
หลักการ: ถ้า g(a) กับ g(b) คนละเครื่องหมาย → มีรากระหว่างนั้น → แบ่งครึ่งไปเรื่อยๆ
```
ทุกรอบ ช่วงแคบลงครึ่งหนึ่ง → หลัง n รอบ ความคลาด ≤ (b−a)/2^n
```
50 รอบ = แม่นถึง 2⁻⁵⁰ ของช่วง (มากเกินพอ)
> **ทำไมเลือก bisection ไม่ใช่ Newton?** bisection *รับประกันลู่เข้า* ถ้ามีรากในช่วง (robust)
> ส่วน Newton เร็วกว่าแต่พังได้ถ้าอนุพันธ์ใกล้ศูนย์ — **เราเลือกความ "ชัวร์" มากกว่าความเร็ว** (ตรงกับ SAFE+STABLE)

### 5.6 เลขคณิตมอดุลาร์ของมุม (Modular arithmetic)
มุมต้องอยู่ในช่วง [0, 2π) เสมอ ใช้ floor-mod (ไม่ใช่ลบทีละ 2π):
```
normalize(r)   = r − 2π * floor(r / 2π)            # ได้ผลในช่วง [0, 2π) เสมอ
angleDiff(a,b) = ((a − b + π) mod 2π) − π          # ผลต่างมุมในช่วง (−π, π]
```
> เกร็ดข้ามภาษา: ตัวดำเนินการ `mod`/`%` ของหลายภาษาคืนค่า *ติดลบ* ได้ และ `Mod` ใน VBA
> ใช้ได้กับจำนวนเต็มเท่านั้น เราจึงต้องเขียน `ModReal_` เองด้วย `Int()` (ซึ่ง floor ลงสู่ −∞)

### 5.7 พาราโบลาของโค้งดิ่ง + ฟังก์ชันเกลี่ย (Parabola & Blending)
- **โค้งดิ่งสมมาตร** เป็นพาราโบลา: ระดับ = ระดับฐาน + เกรด·x + (g₂−g₁)/(2L)·x² (เกรดเป็น % จึงมี /100 ในสูตรจริง)
- **โค้งดิ่งอสมมาตร** (L₁≠L₂): ออฟเซ็ตกลาง `e = (L1·L2)/(2·Ltot)·(g2−g1)/100`
- **ฟังก์ชันเกลี่ยค่า (interpolation/blending):**
  - เชิงเส้น (Linear, ชนิด V): f(t) = t
  - **Smoothstep (ชนิด S):** f(t) = 3t² − 2t³ → ที่ปลายทั้งสอง f′=0 (เปลี่ยนค่านุ่มนวล ไม่กระชาก)
  ใช้ในการยกโค้ง (cross-fall) ให้ค่อยๆ เอียง

### 5.8 สรุปแผนที่คณิตศาสตร์ → จุดใช้งาน
```
ตรีโกณ/พิกัด ........... forwardCompute, inverseCompute, azimuthFromCoords
เรขาคณิตวงกลม ......... pointOnElement (วงกลม), หา center, คอร์ด
พีชคณิตเชิงเส้น ......... การหมุน spiral, dot product แยก along/offset
แคลคูลัส/ความโค้ง ...... θ(s)=∫k, รูปร่างโค้งคลายตัว 4 ชนิด
ซิมป์สัน .............. ตำแหน่งปลาย spiral
แบ่งครึ่ง (bisection) ... inverse บน spiral (หาเท้าตั้งฉาก)
มอดุลาร์มุม ............ normalizeAngle, angleDiff
พาราโบลา/smoothstep ... โค้งดิ่ง, การยกโค้ง cross-fall
```
