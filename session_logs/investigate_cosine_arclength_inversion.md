# สืบสวน COSINE arclength inversion (Simpson s(a) + bisection) — 2026-07-11

ต่อยอดจาก `session_logs/investigate_cosine_totaly_theta_export.md` (2026-07-07) ที่สรุปว่า
theta/totalY/tanShort ของ COSINE ยังใช้ค่าประมาณ x≈s (คลาดเคลื่อน 0.00178°–0.01029° เทียบ
Civil 3D จริง) และต้องแก้ core engine ไม่ใช่ export-only patch

ผู้ใช้ขอให้สืบสวนแนวทางที่ออกแบบไว้: แทนที่จะสมมติ `a = d/X` ตรงๆ ให้แก้สมการอินทิกรัล
ความยาวส่วนโค้งจริง `s(a) = ∫₀ᵃ X·√(1+(dy/dx)²) da'` แล้ว bisection หา `a` ที่ `s(a)=d`
**นี่เป็นรายงานสืบสวนเท่านั้น รันเป็นสคริปต์แยกใน scratchpad ยังไม่แก้โค้ดใน `src/smt/` ใดๆ**

สคริปต์ที่ใช้: `investigate_arclength_inversion.py` (scratchpad, ไม่ commit)

## 1. อนุพันธ์ dy/dx ที่ a — ยืนยันแล้วตรงกับ argument ของ atan() ในสูตร theta ที่มีอยู่

ไม่มี sympy ติดตั้งในเครื่องนี้ (`ModuleNotFoundError`) จึงหาด้วยมือแล้วยืนยันเชิงตัวเลข

หาด้วยมือ จาก `y(a) = X²/R · (a²/4 - (1-cos(πa))/(2π²))`, `x(a) = a·X`:

```
dy/da = X²/R · (a/2 - sin(πa)/(2π))
dx/da = X
dy/dx = (dy/da)/(dx/da) = X/R · (a/2 - sin(πa)/(2π))
```

ตรงกับ argument ของ `atan(...)` ใน `theta(a) = atan(X/R · (a/2 - sin(πa)/(2π)))` ที่มีอยู่แล้ว
ใน `_sine_halfwave_point` **เป๊ะ** — สมเหตุสมผลเพราะ `tan(theta) = dy/dx` ตามนิยามมุมสัมผัส

ยืนยันด้วย central finite-difference (h=1e-6) ที่ a = 0.1, 0.5, 0.9, 1.0 ทั้งสองกรณี ground truth:

## 2. ฟังก์ชัน s(a) ด้วย Simpson 48 ช่วง — เขียนแล้วใน scratchpad script

```python
def s_of_a(a_target, X, R, n_seg=48):
    h = a_target / n_seg
    total = 0.0
    for i in range(n_seg + 1):
        a = i * h
        integrand = X * math.sqrt(1 + dydx_closed_form(a, X, R) ** 2)
        w = 1 if (i == 0 or i == n_seg) else (4 if i % 2 == 1 else 2)
        total += w * integrand
    return total * h / 3
```

## 3. s(1) เทียบกับ L — **s(1) ไม่เท่ากับ L เป๊ะ ไม่ผ่านเกณฑ์ floating-point noise ที่ตั้งไว้แต่แรก**

| กรณี | X (closed-form) | s(1) (Simpson 48 ช่วง) | L | s(1) − L |
|---|---|---|---|---|
| R=900, L=100 | 99.97201364851851 | 99.9999642132682 | 100.0 | **-3.578673e-05 m** |
| R=250, L=50 | 49.9546621106 | 49.99981260855977 | 50.0 | **-1.873914e-04 m** |

**ข้อเท็จจริงตรงไปตรงมา**: ทั้งสองกรณี `s(1) ≠ L` — ผลต่างมากกว่าเกณฑ์ floating-point noise
ที่กำหนดไว้ล่วงหน้า (1e-6 m) ประมาณ 36 เท่า และ 187 เท่าตามลำดับ **ไม่เข้าเกณฑ์ "ปิดสนิท" ที่
ตั้งไว้แต่แรก** — ตามกฎ ห้ามสรุปเดาว่าเล็กพอ จึงตรวจสอบเพิ่มว่านี่เป็น Simpson quadrature error
(ลดลงได้ถ้าเพิ่มจำนวนช่วง) หรือเป็นความไม่สอดคล้องจริงของสูตร โดยเพิ่มจำนวนช่วง Simpson จาก 48
ไปจนถึง 48,000 (นอกเหนือจากที่ระบุในแผนเดิม แต่จำเป็นเพื่อตอบคำถามนี้อย่างซื่อสัตย์):

| n_seg | R=900,L=100: s(1)−L | R=250,L=50: s(1)−L |
|---|---|---|
| 48 | -3.578673180e-05 | -1.873914402e-04 |
| 96 | -3.578372628e-05 | -1.873865183e-04 |
| 480 | -3.578352636e-05 | -1.873861909e-04 |
| 4,800 | -3.578352552e-05 | -1.873861901e-04 |
| 48,000 | -3.578352620e-05 | -1.873861904e-04 |

**ผลต่างคงที่ตั้งแต่ n_seg=480 ขึ้นไป ไม่ลดลงเลยแม้เพิ่มความละเอียดถึง 48,000 ช่วง** — สรุปได้ชัดเจน
ว่า**ไม่ใช่ error จากการอินทิเกรตเชิงตัวเลขของสคริปต์นี้** (ถ้าเป็น quadrature error ค่าจะลดลงตาม
ความละเอียดที่เพิ่มขึ้น แต่ตัวเลขนี้ไม่ลดลง) แต่เป็น**ความไม่สมบูรณ์แบบเล็กน้อยที่แท้จริงของสูตรปิด
X ของ Autodesk เอง** — คือค่าคงที่ `X = L - 0.0226689447·L³/R²` ที่ Autodesk ให้มา ไม่ได้สอดคล้อง
พอดีกับผลอินทิกรัลความยาวส่วนโค้งจริงของ `dy/dx` (closed-form) ของมันเอง ขนาดของความไม่สมบูรณ์
คือ **0.036 มม. (R=900/L=100)** และ **0.187 มม. (R=250/L=50)** — เล็กกว่าระดับมิลลิเมตร (ไม่ต้อง
หยุดรายงานตามเกณฑ์ "≥ มิลลิเมตร") แต่ก็ **ไม่ผ่านเกณฑ์ floating-point noise ที่ตั้งไว้แต่แรก**
ต้องบันทึกไว้ตรงตามนี้

## 4. Bisection หา a ย้อนกลับจาก d=length — ใกล้เคียง a=1 มาก แต่ไม่เท่ากันพอดี (สอดคล้องกับข้อ 3)

| กรณี | a (bisection, s(a)=L) | a − 1.0 | theta(a_bisect) | theta_exact (a=1, atan(X/2R)) | theta diff |
|---|---|---|---|---|---|
| R=900,L=100 | 1.0000003574166643 | 3.574e-07 | 3.1789442946424464° | 3.1789420268894153° | **2.27e-06°** |
| R=250,L=50 | 1.0000037326469826 | 3.733e-06 | 5.705491502751442° | 5.705449190907088° | **4.23e-05°** |

| กรณี | y(a_bisect) | y_exact (0.14867881635766·X²/R) | y diff |
|---|---|---|---|
| R=900,L=100 | 1.6510643006539305 | 1.6510623161163274 | **1.985e-06 m** |
| R=250,L=50 | 1.4841117020091206 | 1.4840930725353705 | **1.863e-05 m** |

ไม่ตรงกันพอดีในระดับ floating-point precision ตามที่คาดไว้ในแผน (ถ้าข้อ 3 ผ่านเกณฑ์ noise จริง
a_bisect ควรเข้าใกล้ 1.0 ถึงระดับ 1e-12 ขึ้นไป) — สาเหตุตรงกับข้อ 3: เพราะ `s(1) ≠ L` พอดี
(residual จริง 0.036–0.187 มม.) การ bisection หา a ที่ทำให้ `s(a) = L` (arc length เป้าหมายที่
แท้จริงของ element) จึงต้องขยับ a ออกจาก 1.0 เล็กน้อยเพื่อชดเชย residual นั้น เป็นผลลัพธ์ที่
สอดคล้องกันเอง ไม่ใช่ข้อขัดแย้งใหม่

## สรุป

**ข้อ 3 — ต้องบันทึกตรงไปตรงมา**: `s(1) ≠ L` เป๊ะ มี residual คงที่ 0.036 มม. (R=900/L=100) และ
0.187 มม. (R=250/L=50) ที่**ไม่ลดลงเลย**แม้เพิ่มความละเอียด Simpson ถึง 48,000 ช่วง — ยืนยันว่า
เป็นความไม่สมบูรณ์แบบเล็กน้อยที่แท้จริงของสูตรปิด X ของ Autodesk เอง ไม่ใช่ error จากการอินทิเกรต
เชิงตัวเลข **ไม่เข้าเกณฑ์ floating-point-noise ที่ตั้งไว้แต่แรก (<1e-6 m)**

**ข้อ 4 — ผลลัพธ์ดีกว่าที่คาด แม้ข้อ 3 จะไม่ผ่านเกณฑ์เป๊ะ**: การ bisection แก้ย้อนกลับหา `a` จาก
`s(a)=d` จริง ให้ theta ที่ d=length ใกล้เคียงกับค่าปิด a=1 มากในระดับ **0.000002–0.00007 องศา**
(2.27e-06° ถึง 4.23e-05° ในสองกรณี ground truth) — **น้อยกว่าความคลาดเคลื่อนของค่าประมาณ x≈s
ปัจจุบันหลายพันเท่า** (ค่าประมาณปัจจุบันคลาดเคลื่อน 0.00178°–0.01029° ในกรณีเดียวกัน, และ
0.003546624° ที่ R=500/L=70 เทียบ bisection ที่เหลือแค่ 7.178e-06° — ต่างกันประมาณ 494 เท่า)
**สรุปว่าแนวทาง Simpson s(a) + bisection นี้ validate แล้วว่าใช้ได้จริงในทางปฏิบัติ** — แม้จะไม่
บรรลุความแม่นยำระดับ floating-point ต่อสูตรปิด a=1 พอดี (เพราะ residual ในตัวสูตรปิด X เอง
ตามข้อ 3) แต่ลดความคลาดเคลื่อนจากค่าประมาณปัจจุบันลงอย่างมีนัยสำคัญในทุกกรณีที่ทดสอบ (3 กรณี:
R=900/L=100, R=250/L=50, R=500/L=70)

ข้อสังเกตสำหรับการวางแผนแก้ในรอบถัดไป (ยังไม่ใช่แผนแก้ เป็นข้อสังเกตปิดท้ายเท่านั้น):
1. residual ที่พบในข้อ 3 หมายความว่า Civil 3D อาจไม่ได้นิยาม X จาก
   `∫₀ᴸ √(1+(dy/dx)²) dx = L` พอดีตามที่แผนเดิมสันนิษฐาน — เอกสารที่มีไม่ได้ระบุที่มาของสูตร
   อินทิกรัลนี้ตรงๆ
2. ถ้าจะใช้แนวทาง bisection จริง ต้องตัดสินใจว่าจะ bisection หา a ที่ `s(a) = d` (ให้พิกัด/theta
   สอดคล้องกับ arc-length ที่ป้อนเข้าจริง) หรือจะ "บังคับ" a=1 พอดีที่ d=length เพื่อให้ตรงกับ
   `theta_exact = atan(X/2R)` ที่ Civil 3D ground truth ยืนยันไว้แล้ว — สองทางเลือกต่างกันเล็กน้อย
   (ระดับ 1e-6 ถึง 4e-5 องศา ตามข้อ 4) เพราะ residual ในข้อ 3 ทั้งสองทางดีกว่าค่าประมาณปัจจุบันมาก
   แต่การเลือกเป็นการตัดสินใจเชิงนโยบายที่ต้องรอแผนแก้แยกต่างหาก
3. ยังไม่ได้ตรวจ jump discontinuity ที่ขอบ d=length ระหว่างจุดกลางโค้ง (bisection, a<1) กับจุดปลาย
   — เป็นคำถามเปิดเดิมจาก `investigate_cosine_totaly_theta_export.md` ข้อ 5 ที่ยังไม่ได้แก้ในรอบนี้
