## 3. ฟังก์ชันที่จะเพิ่ม

```python
# src/smt/builders/table_splitter.py
_VERTEX_POINT_RE = re.compile(r'^(BP|PI-\d+|EP)$')

def split_mixed_alignment_table(
    rows: list[list[Any]],
) -> tuple[list[list[Any]], list[dict[str, Any]]]:
    """แยกตารางดิบที่มี BP/PI-n/PT/PC/TS/SC/CS/ST/EP ปนกัน ออกเป็น (vertex_rows, drawing).

    vertex_rows : [header] + แถวที่ POINT ตรง regex ^(BP|PI-\\d+|EP)$ หรือแถว POINT ว่าง
                  (compound sub-row ที่ parse_pi_table ต้องการติดกับ PI ก่อนหน้า) —
                  ป้อนต่อ parse_pi_table() ได้ตรง ๆ
    drawing     : list ของ {'name','sta','n','e'} จากแถวที่เหลือทั้งหมดที่ไม่ว่างและไม่ตรง
                  regex ด้านบน (PT/PC/TS/SC/CS/ST) — ป้อนต่อ check_against_drawing() ได้ตรง ๆ

    ตัวเลขในคอลัมน์ sta/northing/easting/radius/ls/lsin/lsout/delta จะถูกตัด thousands-
    separator comma ออกก่อน (เช่น "1,537,772.85" -> "1537772.85") เพราะ csv.reader คืนค่า
    เป็น string ที่มี comma อยู่ในเซลล์เดียว (quoted field) และ parse_pi_table()/float()
    เดิมไม่รองรับ comma นี้ — การตัด comma ทำใน splitter นี้เท่านั้น ไม่แตะ parse_pi_table()
    """
```

หมายเหตุออกแบบ:
- แถว POINT ว่าง (blank) + มีค่า RADIUS → เป็น compound sub-row ที่ `parse_pi_table()`
  ผูกกับ PI ก่อนหน้า ต้องส่งเข้า `vertex_rows` ไม่ใช่ `drawing` (แม้ผู้ใช้ระบุ regex เฉพาะ
  BP/PI-n/EP ไว้ แต่ compound sub-row ไม่มีชื่อจุด จึงเข้า `drawing` ไม่ได้อยู่แล้ว — ป้องกัน
  ไว้เผื่อไฟล์อื่นในอนาคตมี compound sub-row; `HOR_ORR_04.csv` ไม่มีแถวแบบนี้เลย จึงไม่กระทบ
  ผลลัพธ์ปัจจุบัน)
- แถวว่างสนิททุกเซลล์ → ข้าม (ไม่เข้าทั้ง 2 ชุด)
- `drawing` ใช้เงื่อนไข "ไม่ตรง regex vertex" แทนการ hardcode รายชื่อ PT/PC/TS/SC/CS/ST
  6 ตัว เพื่อให้รองรับป้ายชื่ออื่นที่เป็นไปได้ (เช่น SPI ถ้ามีในอนาคต) โดยพฤติกรรมกับ
  `HOR_ORR_04.csv` เหมือนกันทุกประการ (ผลลัพธ์ตรวจแล้วในขั้นตอนที่ 5)

## 7. ความเสี่ยง

- Regex `^(BP|PI-\d+|EP)$` case-sensitive ตามที่ระบุ — ถ้าไฟล์อื่นใช้ตัวเล็ก (bp/pi-1/ep)
  จะไม่ match และตกไปเป็น "จุดที่เหลือ" (drawing) โดยไม่มี error — ยอมรับความเสี่ยงนี้เพราะ
  โจทย์ระบุ regex ตรงตัวและ `HOR_ORR_04.csv` ใช้ตัวใหญ่ทั้งหมด
- การตัด comma ทำแบบ blanket (`str.replace(',', '')`) กับทุกคอลัมน์ตัวเลข — ถ้าในอนาคตมีค่า
  ที่ comma มีความหมายอื่น (ไม่ใช่ thousands separator) จะพัง — ปัจจุบันไม่มีกรณีนี้ในไฟล์จริง
  ที่เจอ
- การแก้ `test_data/HOR_ORR_04.csv` เป็นการแก้ไฟล์ข้อมูลจริงจากงานจริง — จำกัดขอบเขตเฉพาะ
  2 เซลล์ (NORTHING/EASTING แถว PI-10 เท่านั้น) ไม่แตะค่าอื่นในไฟล์แม้แต่ตัวเดียว และมี
  หลักฐาน 2 ชั้นรองรับ (tangent-intersection คำนวณเอง + ผู้ใช้ยืนยันจาก source ต้นทางตรงกัน
  พอดี 1000.000) ความเสี่ยงต่ำ
